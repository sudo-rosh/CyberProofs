# Windows Event Log Investigation
**Date:** March 2026  
**Environment:** Windows 10 VM (VirtualBox)  
**Tools Used:** Windows Event Viewer, PowerShell, CMD  
**Objective:** Understand Windows Security Event Logs and demonstrate 
detection of failed login attempts — a core SOC analyst daily task.

---

## Background

Windows Security Event Logs are one of the primary data sources a SOC 
analyst uses to detect suspicious activity. Every login attempt, process 
creation, and account change generates an Event ID that tells a story. 
This investigation demonstrates how to generate, locate, and interpret 
two of the most critical Event IDs in SOC work.

---

## Key Event IDs Investigated

| Event ID | Description | SOC Relevance |
|---|---|---|
| 4624 | Successful account logon | Baseline normal activity — spikes indicate lateral movement |
| 4625 | Failed account logon | Single failures are normal — repeated failures indicate brute force |
| 4688 | New process created | Used to detect malicious process execution |
| 4720 | User account created | Persistence mechanism — attackers create accounts |
| 4740 | User account locked out | Result of brute force or password spray attack |

---

## Investigation Steps

### Step 1 — Accessed Windows Security Event Log
Opened Event Viewer via:
```
Win + R → eventvwr.msc → Windows Logs → Security
```

### Step 2 — Generated a Failed Login Event (4625)
To deliberately trigger a failed login and observe how it appears 
in logs, ran the following in CMD:
```cmd
runas /user:fakeuser cmd
```
This attempted to run CMD as a non-existent user, generating a 
failed authentication attempt.

### Step 3 — Located the Event in Event Viewer
Filtered Security log by Event ID 4625.

**Findings from the generated event:**

| Field | Value |
|---|---|
| Event ID | 4625 |
| Log | Security |
| Level | Information |
| Account Name | fakeuser |
| Failure Reason | Unknown username or bad password |
| Logon Type | 2 (Interactive) |

### Step 4 — Verified Successful Login Event (4624)
Located Event ID 4624 generated by my own successful login to 
confirm what normal authentication looks like in comparison.

**Key difference between 4624 and 4625:**
- 4624 contains a valid Account Name and a Logon ID
- 4625 contains the attempted username and a Failure Reason field
- In a brute force scenario, you would see dozens of 4625 events 
  from the same source IP in a short timeframe

---

## PowerShell Query Used

Rather than manually scrolling Event Viewer, used PowerShell to 
query failed login events programmatically — the same way a SOC 
analyst would query a SIEM:
```powershell
Get-EventLog -LogName Security -InstanceId 4625 -Newest 20 |
Select-Object TimeGenerated, Message |
Format-List
```

This returns the 20 most recent failed login events with timestamps 
and full message details.

## SOC Detection Scenario

**Scenario:** You are a SOC analyst and your SIEM fires an alert — 
47 Event ID 4625 failures from the same source IP in 3 minutes 
targeting the same account.

**What this indicates:** Password spraying or brute force attack.

**Immediate actions:**
1. Identify the source IP — is it internal or external?
2. Check if any 4624 (successful login) followed the 4625 failures
3. If yes — account may be compromised, escalate immediately
4. If no — block the source IP at the firewall, lock the targeted account
5. Check Event ID 4740 — was the account locked out automatically?
6. Document timeline and escalate to Tier 2 if successful login detected

---

## Key Takeaways

- Windows Event Logs are generated continuously and contain detailed 
  authentication, process, and account activity
- A single 4625 event is noise. A pattern of 4625 events is a signal.
- PowerShell allows programmatic log querying — essential for 
  automation and SIEM integration
- Understanding what normal looks like (4624) makes anomalies (4625 
  spikes) immediately obvious
